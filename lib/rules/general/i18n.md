# Internationalization (i18n) Guidelines

> **Scope**: Baseline i18n for ALL projects. Framework-specific rules take precedence.

## CRITICAL REQUIREMENTS

> **ALWAYS**: Externalize ALL user-facing strings into locale/translation files
> **ALWAYS**: Use semantic key names (`common.save`, not `"Save"` as key)
> **ALWAYS**: Support at least two languages to validate the i18n setup works
> **ALWAYS**: Use interpolation for dynamic values (`"Hello, {name}"`, not string concatenation)
> **ALWAYS**: Handle pluralization via the i18n framework (not manual if/else)
>
> **NEVER**: Hardcode user-facing strings in templates or code
> **NEVER**: Use literal text as translation keys
> **NEVER**: Concatenate translated strings (word order differs across languages)
> **NEVER**: Forget to add keys to ALL supported locale files

## 1. Key Naming

```
feature.section.element

✅ GOOD:
  common.save          → "Save" / "Speichern"
  pharmacy.form.name   → "Pharmacy Name" / "Apothekenname"
  validation.email     → "Invalid email" / "Ungültige E-Mail"

❌ BAD:
  Save                 → Literal text as key
  pharmacyNameLabel    → Flat, no hierarchy
  error1               → Meaningless key
```

### Key Hierarchy

```json
{
  "common": { "save": "Save", "cancel": "Cancel", "delete": "Delete", "loading": "Loading..." },
  "validation": { "required": "This field is required", "email": "Invalid email address" },
  "pharmacy": { "title": "Pharmacy", "list": "Pharmacy List" }
}
```

## 2. File Structure

```
locales/
├── en/
│   ├── common.json
│   ├── validation.json
│   └── {feature}.json
├── de/
│   ├── common.json
│   ├── validation.json
│   └── {feature}.json
└── index.ts
```

> **ALWAYS**: One file per feature/domain, mirrored across all locales.
> **ALWAYS**: Group shared strings in `common.json`.

## 3. Interpolation & Pluralization

❌ BAD: String concatenation (word order breaks in other languages)

```json
{ "welcome": "Welcome" }
```

Then in code: `t('welcome') + ", " + name + "!"` — breaks when word order differs.

✅ GOOD: Interpolation

```json
{ "welcome": "Welcome, {name}!" }
```

✅ GOOD: Pluralization via i18n framework

```json
{ "items": "No items | One item | {count} items" }
```

## 4. Formatting

> **ALWAYS**: Use locale-aware formatting for dates, numbers, and currencies
> **NEVER**: Hardcode date formats (`MM/DD/YYYY` vs `DD.MM.YYYY`)

| Data | Approach | Example |
|---|---|---|
| Dates | `Intl.DateTimeFormat` or i18n library | `25.03.2026` (de) / `03/25/2026` (en) |
| Numbers | `Intl.NumberFormat` | `1.234,56` (de) / `1,234.56` (en) |
| Currency | `Intl.NumberFormat` with currency | `1.234,56 €` (de) / `€1,234.56` (en) |

## 5. Backend i18n

> **ALWAYS**: Accept `Accept-Language` header or user preference for API responses
> **ALWAYS**: Return error messages as translation keys (not translated text) for API clients
> **ALWAYS**: Store user locale preference in user profile/settings

**✅ GOOD**: API returns key, client translates:

```json
{ "error": { "code": "validation.email_invalid", "field": "email" } }
```

**❌ BAD**: API returns pre-translated text (breaks if client has different locale):

```json
{ "error": "Ungültige E-Mail-Adresse" }
```

## 6. Anti-Patterns

| Anti-Pattern | Do Instead |
|---|---|
| Hardcoded strings in templates | Use `t('key')` / translation function |
| Missing translation in one locale | Add key to ALL locale files |
| String concatenation for sentences | Use interpolation (`{name}`, `{count}`) |
| Duplicating translations across files | Share via `common.json` |
| `if (count === 1)` for plurals | Use i18n pluralization rules |
| Hardcoded date format `MM/DD/YYYY` | Use `Intl.DateTimeFormat` or i18n library |
